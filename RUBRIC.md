# RVA Hacks: Unified Judging Rubric

**How this document relates to pillar materials:** Judges use **this** file for the six scoring categories, the 1-5 scale, and the judge-facing prompts in each category section. Pillar repositories (for example `CHALLENGE.md` and any printable handouts) may include **optional** examples to help **participants** think about impact, users, feasibility, and equity for a specific problem statement. Those examples **do not replace, override, or limit** how judges apply this rubric during scoring.

## Overall design principle

One common scoring framework across all awards. Judges do not relearn a new system three times. Only the **weights** change to reflect the purpose of each award.

Three benefits:

1. **Consistency:** Judges evaluate the same dimensions every time.
2. **Fairness:** Teams are not judged on completely different hidden standards.
3. **Clarity:** The difference between awards is explicit in the weighting, not buried in vague instructions.

This fits the event well because the hackathon is organized around the Mayor's Action Plan pillars and track-based judging, while also making room for both implementation-minded and visionary solutions. The MAP emphasizes measurable progress, service quality, equity, and partnership: the right foundations for judging civic work.

# The six scoring categories

Each project is scored on the same six categories using a **1-5 scale**.

| Score | Meaning |
|-------|---------|
| 1 | Weak |
| 2 | Emerging |
| 3 | Solid |
| 4 | Strong |
| 5 | Exceptional |

Score each category first, then apply the weight for the relevant award.

---

## Category 1: Impact on the targeted civic problem

This asks whether the project addresses a real public problem in a meaningful way.

For pillar awards, this means impact on the targeted pillar.
For Mayor's Choice and Moonshot, this means impact on the city or a major civic challenge overall.

**Judges should look for:**
- Clear statement of the problem
- Strong alignment to a MAP pillar, goal, metric, or pain point
- Likelihood of improving outcomes for residents, staff, or partners
- Evidence that the team understands the underlying issue, not just the interface

**High score looks like:**
- The project clearly solves a meaningful civic problem
- It directly supports a pillar goal or measurable outcome
- The connection between solution and public value is obvious

**Low score looks like:**
- The project is only loosely connected to civic needs
- The problem is vague or generic
- The team built something interesting but not especially useful

---

## Category 2: Feasibility / implementability

This asks whether the solution could realistically be piloted, adopted, or integrated.

This category matters most for Mayor's Choice.

**Judges should look for:**
- Realistic technical scope
- Use of accessible or reasonably obtainable data
- Thoughtful assumptions about city systems, staff capacity, and procurement reality
- A believable path to pilot or implementation

**High score looks like:**
- Could be piloted in the near term
- Uses tools, workflows, and data that plausibly exist
- Team understands implementation constraints

**Low score looks like:**
- Depends on nonexistent data or perfect integrations
- Requires major policy change, budget, or procurement with no plan
- "The City can just connect all departments" nonsense

---

## Category 3: User value

This asks whether the project creates meaningful value for actual people: residents, city employees, nonprofit partners, or community groups.

**Judges should look for:**
- Clear user definition
- Strong understanding of pain points
- Reduction in friction, confusion, delay, or inequity
- Practical usefulness

**High score looks like:**
- Specific user group is identified
- Value proposition is obvious and relevant
- Improves the experience of using, delivering, or accessing a service

**Low score looks like:**
- User is abstract or undefined
- Value is theoretical
- Solution seems built for judges, not users

---

## Category 4: Execution / prototype quality

This asks how well the team turned the idea into something tangible. Not just "how pretty is the demo," but whether the team can show the solution works coherently.

**Judges should look for:**
- Working prototype, not just slides
- Clear flow from problem to interaction to output
- Competent design and technical execution
- Demo credibility

**High score looks like:**
- Demo works
- Flow is understandable
- Prototype supports the team's claims

**Low score looks like:**
- Broken or unclear demo
- Mostly conceptual
- Hard to tell what was actually built

---

## Category 5: Innovation / originality

This asks whether the idea is fresh, ambitious, or creatively insightful.

This category matters most for Moonshot.

**Judges should look for:**
- Original framing of the problem
- Novel use of technology, policy logic, design, or data
- Non-obvious insight
- Big upside if pursued further

**High score looks like:**
- Not just a small variation on a common app
- Reframes the problem in a useful way
- Suggests new long-term possibilities for the City or community

**Low score looks like:**
- Generic chatbot, dashboard, or form wrapper
- Little originality beyond implementation
- "AI but civic"

---

## Category 6: Equity and inclusion

This asks whether the project supports fairer outcomes and broader access. The MAP explicitly centers equity, inclusion, language access, rights protection, and reducing barriers across multiple pillars.

**Judges should look for:**
- Attention to underserved or burdened groups
- Accessibility and usability considerations
- Reduction of barriers to access
- Avoidance of harm, exclusion, or bias

**High score looks like:**
- Team clearly considered who gets left out
- Solution improves access or fairness
- Project is designed to work for more than the easiest users

**Low score looks like:**
- Equity is ignored
- Solution assumes high trust, high literacy, or high digital access
- Could unintentionally widen disparities

# Award 1: Mayor's Choice Award

## Intent

The project with the strongest case for **near-term implementation and immediate civic value**.

This is the "most likely to be piloted soon" award. It favors useful, grounded, implementable solutions over flashy ambition.

## Weighting

| Category | Weight |
|----------|-------:|
| **Impact** | **5** |
| **Feasibility** | **5** |
| User Value | 4 |
| Execution | 3 |
| Equity & Inclusion | 3 |
| Innovation | 1 |

**Maximum possible score:** 105

## Why these weights work

- **Impact** matters because it must solve a real city problem
- **Feasibility** matches Impact because the whole point is near-term usefulness
- **User value** matters because civic systems only matter if they help people
- **Execution** matters, but less than actual implementability
- **Innovation** matters least: this award is for the most adoptable idea, not the coolest

## What should win

- Could plausibly be piloted in the next 3-12 months
- Improves service delivery, staff workflow, resident experience, or access
- Works with realistic assumptions about data and systems
- Creates immediate public value

## What should not win

- Inspiring but nowhere near implementable
- Needs massive systems integration
- Depends on ideal data that probably does not exist
- Mostly visionary decks with a demo shell

## Judge prompt

> If the City wanted to test one project soon, with limited time and attention, which one would be easiest to move forward while still creating real public value?

## Tie-breaker sequence

1. Higher Feasibility
2. Higher Impact
3. Higher User Value

# Award 2: Moonshot Award

## Intent

The project with the strongest **long-term upside, originality, and transformative potential**.

This is the "most visionary and promising over time" award. It rewards bold thinking without requiring near-term implementation readiness.

## Weighting

| Category | Weight |
|----------|-------:|
| **Innovation** | **5** |
| Impact | 4 |
| User Value | 3 |
| Equity & Inclusion | 3 |
| Feasibility | 2 |
| Execution | 2 |

**Maximum possible score:** 95

## Why these weights work

- **Innovation** carries the most weight: the whole point is original thinking
- **Impact** remains high because this still needs to matter civically
- **User value** and **equity** keep the award from turning into gadget worship
- **Feasibility** matters only enough to weed out total fantasy
- **Execution** matters least: a moonshot can be rough and still be profound

## What should win

- Introduces a bold or novel civic concept
- Reframes a hard problem in a compelling way
- Could meaningfully shape the city over a longer horizon
- Shows strategic imagination, not just polish

## What should not win

- Only technically impressive
- Totally disconnected from a real civic problem
- Pure science fiction with no credible logic
- Incremental but dressed up with buzzwords

## Judge prompt

> Which project, if nurtured over time, could most meaningfully change how Richmond addresses a civic challenge?

## Tie-breaker sequence

1. Higher Innovation
2. Higher Impact
3. Higher Equity & Inclusion

# Award 3: Pillar-Specific Awards (7 total)

## Intent

The strongest project within **the targeted pillar**.

Unlike Mayor's Choice and Moonshot, pillar awards balance practicality and originality. You want the best answer to that pillar's problem, not necessarily the most implementable overall or the most visionary overall.

## Weighting

| Category | Weight |
|----------|-------:|
| **Impact on the targeted pillar** | **5** |
| User Value | 4 |
| Feasibility | 3 |
| Innovation | 3 |
| Execution | 3 |
| Equity & Inclusion | 3 |

**Maximum possible score:** 105

## Why these weights work

- Strongly prioritizes **pillar relevance and impact**
- Still cares a lot about whether the solution helps users
- Balances both **feasibility** and **innovation** equally; neither dominates
- Creates a middle path: more grounded than Moonshot, less implementation-biased than Mayor's Choice

## What should win

- Best addresses the targeted pillar's problem
- Understands the context of the issue
- Offers meaningful public value
- Is both promising and credible

## What should not win

- Only matches the pillar loosely
- Good product but weak civic fit
- Overbuilt demo with thin relevance
- Clever but not responsive to the actual challenge

# Scoring sheet

For each project, judges score 1-5 per category and write one sentence of notes.

| Category | Score (1-5) | Notes |
|----------|------------:|-------|
| Impact | | |
| Feasibility | | |
| User Value | | |
| Execution | | |
| Innovation | | |
| Equity & Inclusion | | |

Then multiply each score by the weight for the relevant award and sum.

**Final weighted score = Σ (category score × category weight)**

This gives a transparent final number while preserving the same judging language across all three awards.
