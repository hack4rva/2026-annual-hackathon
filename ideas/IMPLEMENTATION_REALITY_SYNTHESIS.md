## Five Structural Walls

These are the infrastructure constraints that separate ideas that can ship from ideas that can't. No amount of engineering bypasses them.

### 1. The 311 / AvePoint Wall
**Systems:** RVA 311 (AvePoint on Dynamics 365), BizTalk middleware, Cityworks
**Impact:** Blocks Text 311, RVA Help, and any idea requiring programmatic 311 submission or status reading
**Ideas affected:** 3 directly, 2 indirectly (Hey804, Service Wizard work around it)
**Workaround:** Treat 311 as a destination (link to web forms/phone numbers), not a dependency

### 2. The OnBase / RAPIDS Wall
**Systems:** Oracle E-Business Suite (RAPIDS), OnBase ECM (contract PDFs)
**Impact:** Blocks Mira, CivicPulse AI, RVA Contract Lens, Vendor Contract Mgmt
**Ideas affected:** 4 directly
**Workaround:** Use Socrata contract metadata (structured, public) instead of OnBase PDFs (rich, locked)

### 3. The EnerGov Wall
**Systems:** Tyler EnerGov (permits, compliance tracking)
**Impact:** Blocks Affordable Housing Dashboard, iBuild, SIMBY (data source), ADA Detour Publisher
**Ideas affected:** 4 directly
**Note:** Replacement RFP is in progress — the next system may have an API. Time this, don't fight it.

### 4. The B2GNow Wall
**Systems:** B2GNow vendor directory (MBE certifications, capabilities)
**Impact:** Blocks OMBD Proactive Match Dashboard — the idea with the strongest ownership (5/5) in Economy
**Ideas affected:** 1 directly, but it's the highest-impact PS1 idea in the pillar
**Workaround:** Request a scheduled CSV export from OMBD; even monthly would unlock the core matching logic

### 5. The DPW Fleet GPS Wall
**Systems:** DPW vehicle telemetry (GPS, zone completion, plow positions)
**Impact:** Blocks All Your Bass, Road Maintenance Fleet Tracker, partially limits GRTC Conflict Detector
**Ideas affected:** 3 directly
**Workaround:** Zone-level pass/complete status (not vehicle GPS) would unblock the resident-facing use case

## Hidden Connections — Ideas That Share Infrastructure

These ideas share data sources, users, or architecture and should be built as connected investments, not isolated projects.

### Bundle 1: Socrata Contracts Pipeline
**Shared data:** City Contracts `xqn7-jvv2`, Payment Register `5y73-enav`
**Ideas:** Contract Expiry Dashboard → Contract Expiry Alerts → Contract-Payment Drift Scanner → Richmond Spend Mirror → Richmond Contract Navigator
**Combined score:** 109/125 (avg 21.8)
**Insight:** Build the Socrata data pipeline once, serve five products. The dashboard is the foundation; alerts add push delivery; drift scanner adds audit intelligence; Spend Mirror reframes for MBE vendors; Contract Navigator adds AI matching. Each is independently valuable; together they're a procurement intelligence stack.

### Bundle 2: Legistar Development Pipeline
**Shared data:** `webapi.legistar.com/v1/richmondva/` (Matters, Events, Bodies)
**Ideas:** Site-Sign Text-Back → RVA Plotlines/Dev Tracker → Public Comment Atlas → Participation Analytics Dashboard
**Combined score:** 78/100 (avg 19.5)
**Insight:** Legistar is the most underused high-quality API in Richmond's stack. Four ideas independently discovered it. A shared Legistar-to-GeoJSON ETL (matters → addresses → map pins) would serve all four. The Text-Back SMS service is the fastest proof-of-concept.

### Bundle 3: GeoHub Infrastructure Lookup
**Shared data:** ArcGIS REST (CIP, parcels, assessor, zoning, neighborhoods)
**Ideas:** StreetLights → "What's On My Street?" → Snow Tier Compass → HUD Rent-Cap Validator → Subsidy & Covenant Snapshot
**Combined score:** 103/125 (avg 20.6)
**Insight:** StreetLights already proved the GeoHub CIP pattern works. "What's On My Street?" extends it with multi-source aggregation. Snow Tier Compass uses the same geocoder + polygon lookup pattern for a different dataset. The housing ideas use parcels/assessor layers. Shared geocoding and address-to-polygon infrastructure benefits all five.

### Bundle 4: Client-Side PDF / Zero-Server Tools
**Shared architecture:** Browser-based form → PDF generation, no backend, no data storage
**Ideas:** Core Eligibility Pack Builder, Cross-Agency Handoff Memo, Grant Pack Builder, Storefront Work-Window Brief
**Combined score:** 84/100 (avg 21.0)
**Insight:** Four ideas independently converged on the same architectural pattern: client-side tools that generate printable artifacts without storing any data on a server. This pattern scores highest on Integration Surface and Maintenance Burden because there's nothing to integrate and nothing to maintain. The shared technical pattern (form → validate → render → PDF export) could be a reusable template.

### Bundle 5: GRTC Transit Overlay
**Shared data:** GRTC static GTFS (routes, stops, schedules)
**Ideas:** GRTC Shift-Fit Scout, GRTC–Construction Conflict Detector, Neighborhood Anchor Map, Reentry 72-Hour Navigator
**Combined score:** 74/100 (avg 18.5)
**Insight:** Four ideas across three pillars need "how do I get there by bus?" answered. A shared GTFS-to-nearest-stop-and-trip-time library would serve all four. The Shift-Fit Scout is the simplest proof (address + time → last bus check); the Anchor Map adds geographic context; the Conflict Detector adds spatial join intelligence.

### Bundle 6: SMS/Twilio Channel
**Shared infrastructure:** SMS gateway (Twilio or similar)
**Ideas:** Site-Sign Text-Back, First Job SMS Kit, Weekend Culture SMS, Contract Expiry Alerts (email, upgradeable to SMS)
**Combined score:** 81/100 (avg 20.3)
**Insight:** Four ideas need to send or receive SMS. A shared Twilio account with a known phone number and message routing infrastructure reduces per-idea setup cost and creates a recognizable "City SMS" channel (though none of these are City-operated today).

### Bundle 7: CultureWorks Events Feed
**Shared data:** CultureWorks/Localist ICS, RSS, JSON API
**Ideas:** Weekend Culture SMS, Vecina, What's On RVA
**Combined score:** 54/75 (avg 18.0)
**Insight:** A single CultureWorks partnership agreement would unlock three ideas simultaneously. Vecina already has a working integration. Weekend Culture SMS adds push delivery. What's On RVA adds AI discovery (but at the cost of DIT AI policy review).

## Pattern Analysis

### What scores highest?
**Narrow scope + single public API + zero integration.** The top 10 ideas average 2.3 data sources each. The bottom 10 average 4.7. Ideas that try to aggregate many sources score lower because each source adds maintenance burden and stale-data risk.

### What scores lowest?
**Platforms that require data that doesn't exist.** Every C-tier idea (Text 311, Richmond Stories Online, Community Story Pop-Up Kit, Displacement Memory Archive, Multi-Pillar AI) depends on data that must be created from scratch — either through community organizing (stories), data entry (fleet GPS), or institutional change (311 API).

### Demo vs. AI-generated ideas?
Demos average 14.6; AI-generated ideas average 19.0. This isn't because AI is better at ideation — it's because the AI-generated ideas were designed *after* the tech stack research was complete, with full knowledge of what data exists and what's locked. Hackathon teams built in 48 hours without that knowledge. The fix isn't "better ideas" — it's better pre-hackathon data briefings.

### Which dimensions kill ideas?
**Integration Surface** is the biggest discriminator. Ideas scoring 5 (standalone) average 19.1 total. Ideas scoring 1-2 (deep integration) average 11.4. The 7.7-point gap is larger than any other dimension. Building standalone tools that consume public APIs is the single highest-leverage architectural decision.

### Where does ownership matter most?
**Ownership Clarity** has the lowest average score across all 73 ideas (3.0). Even S-tier ideas average only 3.8 on ownership. This reflects a structural reality: hackathon ideas often solve problems that sit between departments. The ideas with the highest ownership scores (Contract Expiry Dashboard: 5, OMBD Match Dashboard: 5, MAP Tracker: 5) are the ones where a single department can say "this is mine" without a committee meeting.

## The Meta-Lesson

The most implementable civic tech ideas share three properties:

1. **They read public data, not private data.** Socrata, Legistar, GeoHub, HUD, VDH, GRTC GTFS — the public API surface is richer than most teams realized.
2. **They generate artifacts, not integrations.** PDFs, CSVs, maps, SMS messages, email alerts — outputs that work without changing any City system.
3. **They solve one problem for one user.** Contract renewals for procurement officers. Rent-cap validation for housing compliance staff. Bus-to-shift checks for teen job seekers. Narrow beats broad every time.

The ideas that fail are the ones that say "if the City just gave us access to..." — because the City won't, and even if they did, the maintenance burden would kill the project within 90 days.

**Build what you can ship. Ship what one department will use. Let success create demand for the next thing.**
