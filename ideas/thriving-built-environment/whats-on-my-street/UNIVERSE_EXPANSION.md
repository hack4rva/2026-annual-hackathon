# One-Stop Street Insight Engine: Scaling Richmond's Infrastructure Transparency

## 1. EXECUTIVE SUMMARY

The "What's On My Street?" idea fundamentally shifts municipal infrastructure transparency from a fragmented, map-based paradigm to a unified, text-first accessible search engine. Currently, Richmond's infrastructure data is scattered across multiple portals, creating a significant visibility gap for residents. While cities like Philadelphia and Seattle have attempted unified trackers, they often rely on map interfaces that fail Web Content Accessibility Guidelines (WCAG) for screen readers, or they suffer from data blind spots by excluding state or private utility projects. 

To expand this idea's universe beyond a simple city-owned hackathon project, Richmond must adopt a hybrid governance model and leverage regional and federal funding. By positioning this tool as an "advanced traveler information system," the project becomes eligible for significant federal grants like the USDOT SMART program [1] and FHWA ATTAIN program [2]. Partnering with local nonprofits and regional bodies like PlanRVA can transform this from a Richmond-only symptom fix into a scalable, multi-jurisdictional data standard that sets a new national benchmark for civic accessibility.

## 2. EXPANDED OWNERSHIP RECOMMENDATION

**Recommended Model: Hybrid City/Nonprofit Partnership**

The most sustainable path forward is a hybrid ownership model. 
* **What the City Should Own:** The City of Richmond must own the underlying data governance, the centralized data-sharing mandates (similar to Seattle's municipal code requiring project entry [3]), and the nightly ETL pipelines. The city is the only entity with the legal authority to compel private utilities and internal departments to standardize their data feeds.
* **What Partners Should Own:** A civic-tech nonprofit (such as a local Code for America brigade or the Richmond Data Coalition) should own the public-facing, text-first web application. This allows for agile UI/UX iteration, independent accessibility auditing, and community-led feature development without being bottlenecked by municipal procurement cycles.
* **What Should NOT be City-Owned:** The resident-facing presentation layer and the community outreach strategy should not be exclusively city-owned, as historical communication gaps can hinder community trust.

## 5. EXISTING EFFORTS / PRIOR ART

* **Philadelphia (StreetSmartPHL):** 
 * *What was attempted:* An interactive mapping application displaying current/future street closures, paving, and snow plowing info [21]. It aggregates multiple layers via an ArcGIS FeatureServer [22].
 * *What worked:* High frequency of updates and aggregation of 16+ layers [23].
 * *Lesson:* While data aggregation is strong, map-first interfaces often struggle with WCAG compliance. Richmond must prioritize the text-first list view.
* **Seattle (Coordinate / dotMaps):**
 * *What was attempted:* A centralized intake model where agencies performing work in the right of way planned at least 6 months ahead must enter project info into Coordinate [3] [24].
 * *What worked:* Mandated municipal code ensures high data capture, including private utilities.
 * *Lesson:* Richmond needs a policy mandate (like Seattle's SMC 15.32.050) to force data compliance, rather than relying solely on voluntary API scraping.
* **Richmond (GeoHub CIP Dashboard):**
 * *What was attempted:* An interactive data and map tool to explore active capital improvement projects [25].
 * *What worked:* Proved that Richmond can successfully publish and maintain a FeatureLayer for infrastructure [26].
 * *Lesson:* The foundation exists, but it must be expanded beyond just CIP to include VDOT and utility data to be truly useful to residents.

## 6. SCALE SHIFT ANALYSIS

Framing "What's On My Street?" solely as a Richmond project limits its potential. 
* **Regional Shift:** By integrating with regional bodies, the platform can serve the broader commuter base that crosses county lines daily. This makes the project highly competitive for regional USDOT SMART grants [1].
* **Statewide Shift:** Incorporating VDOT FeatureServer data elevates the tool to a statewide right-of-way awareness platform. If successful in Richmond, the underlying open-source code and ETL pipelines could be adopted by VGIN [15] as a standard for other Virginia municipalities.
* **Ownership Shift:** Shifting the resident-facing layer to a nonprofit while the city provides the data API reduces municipal maintenance burdens and allows the UI to evolve at the speed of modern web development, rather than the speed of government procurement.

## 7. RISK AND CONSTRAINT ANALYSIS

* **Data-Sharing Authority:** Without a municipal mandate (similar to Seattle's [3]), private utilities and siloed departments may refuse to provide data, resulting in an incomplete picture for residents.
* **Technical Integration:** Relying on external APIs (like VDOT or scraped DPW pages) introduces the risk of broken pipelines if upstream data structures change.
* **Accessibility Compliance:** Map-based tools inherently struggle with screen readers. Failing to strictly enforce the text-first UI could result in missing the WCAG 2.2 AA target, jeopardizing federal grant eligibility.
* **Grant Dependence:** Relying solely on a single federal grant (like ATTAIN [2]) creates a fiscal cliff. The project must secure recurring local funding (e.g., VGIN data grants [6]) for long-term maintenance.
