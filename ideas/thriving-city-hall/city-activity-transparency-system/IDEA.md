# City Activity Transparency System

**Source:** NEW  
**Pillar:** A Thriving City Hall  
**Problem Statement:** Help residents find the right City service or department quickly, so requests get routed correctly the first time.

---

## Description

**Reframed value proposition:** The system is not only a dashboard — it is a **governance cadence** supported by data. Research contrasts tools embedded in daily operations and executive review (e.g., performance STAT-style programs and public scorecards) with monolithic “smart city” catalogs that fade after grants. For Richmond, the stack is described as **split**: an internal GIS-style common operating picture for operations (where data access allows) and a curated public scorecard for residents and media, with tiered latency (not everything real-time). In one sentence: Richmond residents lack a single place to see what the City is working on, how services perform, and whether neighborhoods are treated equitably.

## Key Approach

- **Pattern A (quick win):** Public KPI scorecard — aggregate 15–20 metrics from existing public sources (Socrata, ArcGIS, department reports) with “last updated” stamps, links to sources, and neighborhood comparison on a few equity-sensitive metrics.
- **Pattern B (stretch):** Internal COP plus public transparency — operational map layers where feeds exist; batch-updated public view; recurring leadership sessions with published takeaways (institutional, not only technical).
- Audience: residents, journalists, Council, organizers — plus internal staff where an internal layer is in scope.
- Scope honestly around structural constraints called out in research (e.g., 311 and fleet data walls): build from public data while documenting gaps.
- Start with governance questions (who meets, how often, what decisions data drives), not only more widgets.
