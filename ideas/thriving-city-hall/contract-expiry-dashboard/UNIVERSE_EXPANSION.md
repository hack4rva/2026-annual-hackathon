# From One-City Dashboard to a Regional Procurement Hub: Governance, Funding, and Partnership Blueprint for Richmond’s Contract-Expiry Tracker

## 1. EXECUTIVE SUMMARY

The Contract Expiry Dashboard represents a rare, high-leverage civic innovation opportunity: Richmond is currently the only confirmed mid-sized U.S. city publishing structured contract expiration dates (`effective_to`) via its Socrata open data portal. While hackathon teams focused on complex AI extraction tools, the actual barrier to procurement transparency is UX, not data. By deploying a simple, filterable dashboard that highlights 30/60/90-day expirations, Richmond can immediately solve a core procurement pain point (JTBD 1) while creating a scalable model for other jurisdictions. 

Expanding this idea's universe means shifting it from a localized hackathon project to a regional and national open-contracting exemplar. Because the underlying data pipeline (`xqn7-jvv2`) is already maintained by the City, the presentation layer can be owned and scaled by a civic-tech nonprofit or regional planning body like PlanRVA. This hybrid approach unlocks federal funding (e.g., USDOT SMART Grants, US Treasury SLFRF) and philanthropic support (Open Contracting Partnership, Community Foundation for a greater Richmond) that the city alone might struggle to secure for a niche IT project. Ultimately, this tool serves as the foundational layer for a broader equity initiative, allowing minority and women-owned business enterprises (MWBEs) to proactively prepare for upcoming contract recompetes.

## 2. EXPANDED OWNERSHIP RECOMMENDATION

**Recommended Model: Hybrid City-Data / Nonprofit-Led Presentation**

* **What the City should own:** The City of Richmond must retain ownership of the underlying data pipeline, the Socrata dataset (`xqn7-jvv2`), and the internal ERP/procurement workflows that generate the `effective_to` field. The City's role is strictly as a reliable data provider and internal adopter of the tool.
* **What partners should own:** A civic-tech nonprofit (e.g., Code for America brigade) or a regional body (e.g., PlanRVA) should own the front-end dashboard, the GitHub Pages deployment, the user interface, and the integration of external equity datasets (like the Virginia SWaM directory). 
* **What should NOT be city-owned:** The City should not own the custom front-end application code or the public-facing advocacy layer. City IT departments are often bottlenecked; requiring them to maintain a custom React/JavaScript dashboard introduces unnecessary maintenance risk. By keeping the presentation layer open-source and community-maintained, the tool can iterate faster and easily be cloned by neighboring municipalities.

## 5. EXISTING EFFORTS / PRIOR ART

* **Tempe, AZ Procurement Contracts Dashboard:** Tempe built a dashboard allowing public access to procurement contracts, including a specific "Contract Expiration Date" field [19] [20]. *Lesson:* Explicitly defining the lifespan of a contract is possible and provides a clear model for Richmond's UX.
* **NYC Checkbook:** Ranked as a top transparency tool, it includes a Contracts Domain with an API that exposes the "End date associated with a contract" [21] [22]. *Lesson:* Large-scale implementations prove the value of API-driven end-date exposure, but Richmond can achieve the core value with a much lighter-weight Socrata integration.
* **Portland, OR Open Contracting Data Standard (OCDS):** Portland publishes procurement data using the OCDS model and provides Power BI dashboards for lookup [23] [24]. *Lesson:* Adopting standard data models (like OCDS) makes the tool easier to scale and share with other cities.
* **Cambridge, MA Open Data:** Cambridge publishes a contracts dataset on Socrata that includes `start_date`, `end_date`, and `renewals_remaining` [25]. *Lesson:* Cambridge is Richmond's closest peer in data readiness; their schema validates Richmond's approach.
* **Virginia eVA System:** The state's marketplace offers public reports and vendor searches [26] [27]. *Lesson:* State systems are often complex and hard to navigate for simple expiration tracking; a localized, focused dashboard fills a gap eVA misses.

## 6. SCALE SHIFT ANALYSIS

Framed solely as a Richmond project, the Contract Expiry Dashboard is a useful but niche administrative tool. However, if the frame shifts:
* **Regional Shift (PlanRVA):** If adopted by PlanRVA [11], the dashboard could aggregate contract expirations across Central Virginia. This allows regional MWBEs to see a unified pipeline of upcoming opportunities, drastically increasing the tool's economic impact and making it highly competitive for federal SMART grants [1].
* **Statewide Shift (Virginia SWaM Integration):** By cross-referencing Richmond's expiring contracts with the Virginia SWaM directory [17], the tool transforms from a compliance tracker into an active economic development engine, alerting certified minority-owned businesses to upcoming recompetes.
* **Nonprofit Ownership:** If a nonprofit owns the resident-facing layer, the City is relieved of UI maintenance. The nonprofit can iterate rapidly, add features like email alerts, and package the codebase as a "Replication Kit" for other mid-sized cities, supported by organizations like the Open Contracting Partnership [10].

## 7. RISK AND CONSTRAINT ANALYSIS

* **Data Quality and Maintenance:** The dashboard relies entirely on the Socrata `xqn7-jvv2` dataset. If the City changes its ERP system or stops updating the `effective_to` field, the dashboard breaks.
* **"Link Farm" Failure Mode:** Previous transparency efforts in other cities failed because they merely linked to PDFs or complex portals without structured, machine-readable end dates. The dashboard must remain strictly focused on sortable, filterable data.
* **Adoption and Trust:** Procurement officers may ignore the tool if it is not embedded in their daily workflow. It must be faster and easier to use than their existing internal spreadsheets.
* **Legal and Procurement Blockers:** If the City attempts to build this internally, it could trigger a lengthy procurement cycle. Deploying it via a civic-tech partner on GitHub Pages bypasses this blocker.
